# Σύστημα Διαχείρισης Κουρείου (Barbershop Management)

Desktop εφαρμογή για τη διαχείριση των καθημερινών λειτουργιών ενός κουρείου:
πελάτες, υπηρεσίες, ραντεβού, πληρωμές και εργαζόμενοι. Αναπτύχθηκε με
**Java + JavaFX** και **PostgreSQL**, ακολουθώντας αρχιτεκτονική
**Model–View–Controller (MVC)**. Λειτουργεί **τοπικά**, χωρίς πρόσβαση μέσω
διαδικτύου, σύμφωνα με τις προδιαγραφές (SRS).

> Μάθημα: Τεχνολογία Λογισμικού — Πανεπιστήμιο Δυτικής Αττικής, Ακ. Έτος 2025–2026

## Αρχιτεκτονική

Η εφαρμογή ακολουθεί **MVC** με επιπλέον διαχωρισμό σε επίπεδα (layers), ώστε η
επιχειρησιακή λογική να είναι ανεξάρτητη από το γραφικό περιβάλλον και να μπορεί
να ελεγχθεί με unit tests:

```
  View (FXML)  ──►  Controller  ──►  Service  ──►  DAO  ──►  PostgreSQL
  (resources/view)  (controller/)   (service/)   (dao/)     (βάση)
```

- **Model** (`model/`): οι οντότητες της εφαρμογής (Customer, Employee,
  Appointment, Service, Payment, User) και τα enums. Απλές κλάσεις δεδομένων.
- **View** (`resources/.../view/`): τα αρχεία `.fxml` που ορίζουν το γραφικό
  περιβάλλον, με μοντέρνο στυλ μέσω `styles.css`.
- **Controller** (`controller/`): συνδέει το View με τη λογική. Δεν περιέχει
  επιχειρησιακούς κανόνες — μόνο χειρισμό του UI.
- **Service** (`service/`): εδώ βρίσκονται **όλοι οι επιχειρησιακοί κανόνες**
  (validation, αποτροπή double-booking, κανόνες διαγραφής, υπολογισμός
  απολαβών, έλεγχος δικαιωμάτων). Είναι το επίπεδο που τεστάρεται.
- **DAO** (`dao/`): Data Access Objects — η μόνη επικοινωνία με τη βάση, μέσω
  JDBC και `PreparedStatement` (προστασία από SQL injection).
- **config/DatabaseConnection**: singleton που διαχειρίζεται τη σύνδεση JDBC.

Αυτός ο διαχωρισμός σημαίνει ότι η λογική (π.χ. ο έλεγχος double-booking) δεν
εξαρτάται από το UI ούτε από συγκεκριμένη οθόνη, και μπορεί να επαναχρησιμοποιηθεί
και να ελεγχθεί ανεξάρτητα.

## Εγκατάσταση

### 1. Δημιουργία βάσης δεδομένων

Μέσω pgAdmin ή γραμμής εντολών:

```sql
CREATE DATABASE barbershop;
```

### 2. Εκτέλεση του schema

Τρέξε το `schema.sql` στη βάση `barbershop` (δημιουργεί πίνακες, enum types, την
επέκταση `btree_gist`, το constraint αποτροπής double-booking, indexes, και
αρχικά δεδομένα — μερικές υπηρεσίες + έναν χρήστη `admin`):

```bash
psql -U postgres -d barbershop -f schema.sql
```

### 3. Ρύθμιση σύνδεσης

> ⚠️ Το `db.properties` **δεν** περιλαμβάνεται στο repository (περιέχει κωδικό
> και εξαιρείται μέσω `.gitignore`). Πρέπει να το δημιουργήσεις χειροκίνητα.

Δημιούργησε το αρχείο `src/main/resources/db.properties`:

```properties
db.url=jdbc:postgresql://localhost:5432/barbershop
db.user=postgres
db.password=Ο_ΚΩΔΙΚΟΣ_ΣΟΥ
```

### 4. Ορισμός κωδικού για τον χρήστη admin

Ο αρχικός `admin` στο `schema.sql` έχει προσωρινό (μη έγκυρο) hash. Παρήγαγε ένα
πραγματικό BCrypt hash τρέχοντας την κλάση `PasswordHashGenerator`:

```bash
mvn -q compile exec:java \
  -Dexec.mainClass=gr.uniwa.barbershop.util.PasswordHashGenerator \
  -Dexec.args=admin123
```

Αντίγραψε το hash που τυπώνεται και τρέξε στη βάση:

```sql
UPDATE users SET password_hash = '<το hash>' WHERE username = 'admin';
```

```bash
mvn javafx:run
```

Σύνδεση με χρήστη `admin` και τον κωδικό που όρισες (π.χ. `admin123`).

## Ρόλοι χρηστών

| Ρόλος | Πρόσβαση |
|------|----------|
| **Διαχειριστής** (Admin) | Πλήρης πρόσβαση, συμπεριλαμβανομένης της διαχείρισης εργαζομένων |
| **Υπάλληλος** (Employee) | Πελάτες, ραντεβού, πληρωμές — **όχι** διαχείριση εργαζομένων |
| **Γραμματέας** (Secretary) | Ραντεβού και καθημερινές λειτουργίες — **όχι** διαχείριση εργαζομένων |

Ο έλεγχος ρόλων εφαρμόζεται σε **δύο επίπεδα**: το UI κρύβει τα μη
επιτρεπόμενα κουμπιά, και το service layer απορρίπτει μη εξουσιοδοτημένες κλήσεις.

## Λειτουργίες

- **Πελάτες** — προσθήκη, επεξεργασία, διαγραφή, αναζήτηση. Καταγραφή
  καθυστερήσεων και audit (ποιος/πότε έκανε την τελευταία αλλαγή).
- **Υπηρεσίες** — διαχείριση υπηρεσιών (όνομα, τιμή, διάρκεια), απενεργοποίηση.
- **Ραντεβού** — δημιουργία/επεξεργασία/ακύρωση. Κάθε ραντεβού συνδέεται με
  πελάτη, εργαζόμενο και υπηρεσία. Η ώρα λήξης υπολογίζεται αυτόματα από τη
  διάρκεια της υπηρεσίας.
- **Ημερήσιο Πρόγραμμα** — προβολή των ραντεβού μιας ημέρας, ταξινομημένα κατά ώρα.
- **Πληρωμές** — καταχώρηση πληρωμών (Μετρητά/Κάρτα), συνδεδεμένων με πελάτη
  και προαιρετικά με συγκεκριμένο ραντεβού.
- **Εργαζόμενοι** (μόνο Admin) — διαχείριση στοιχείων και **αυτόματος υπολογισμός
  απολαβών** (βασικός μισθός + προμήθεια επί των ολοκληρωμένων υπηρεσιών).

## Επιχειρησιακοί κανόνες

- **Αποτροπή διπλής κράτησης (double-booking):** ένας εργαζόμενος δεν μπορεί να
  έχει δύο επικαλυπτόμενα ραντεβού. Ελέγχεται στο `AppointmentService` (φιλικό
  μήνυμα) και επιβάλλεται από constraint `EXCLUDE` στην PostgreSQL (δικλείδα για
  ταυτόχρονες εγγραφές).
- **Διαγραφή πελάτη:** επιτρέπεται μόνο από **Διαχειριστή** και μόνο αν ο πελάτης
  δεν έχει ενεργά (SCHEDULED) ραντεβού.
- **Απολαβές:** υπολογίζονται δυναμικά, δεν αποθηκεύονται.
- **Validation:** έλεγχος υποχρεωτικών πεδίων πριν από κάθε εγγραφή.
- **Κωδικοί:** αποθηκεύονται μόνο ως BCrypt hashes, ποτέ ως απλό κείμενο.

## Unit Tests

Τα tests (JUnit 5) ελέγχουν το **service layer** χωρίς να χρειάζεται βάση
δεδομένων, εφόσον στοχεύουν στην καθαρή λογική:

- `PasswordUtilTest` — κρυπτογράφηση και επαλήθευση κωδικών.
- `EmployeeEarningsTest` — η φόρμουλα υπολογισμού απολαβών.
- `ValidationTest` — κανόνες validation και ελέγχου δικαιωμάτων.
- `ModelTest` — enums και βοηθητικές μέθοδοι μοντέλων.

Εκτέλεση:

```bash
mvn test
```

ή από το IntelliJ: δεξί κλικ στο `src/test/java` → **Run All Tests**.
